# Expand research.py allowlist with academic/data sources and tiered exfil mitigation

## Status

- [x] Phase 1: Externalize seed to `templates/research-allowlist.txt` (mechanical; Haiku ok)
- [x] Phase 2: Rewrite the allowlist body with READ-ONLY + UPLOAD-CAPABLE tiers and expanded scope
- [ ] Phase 3: Add `--reseed-allowlist` flag

## Context

`research.py` ships with a seed allowlist embedded as a Python triple-quoted string at `research.py:174-269` (`_ALLOWLIST_SEED`). On first launch, `seed_allowlist()` (`research.py:272-276`) writes it to `~/.research/allowlist.txt`; the file is then user-editable. The current seed is decent but:

- **Mixes write-capable and read-only hosts without flagging them.** `gitlab.com` (push, MR creation) and `huggingface.co` (model/dataset upload) sit alongside `arxiv.org` and `wikipedia.org`. tinyproxy filters by hostname only — it cannot distinguish read paths from write paths on the same host. The user wants the seed to make this distinction visible.
- **Is narrower than the intended use of this stack.** No data repositories (data.gov, FRED, World Bank), no software/language docs (docs.python.org, readthedocs.io), no standards bodies (w3.org, ietf.org), no legal/regulatory (sec.gov, federalregister.gov, courtlistener), no major universities, no broad science publications (Quanta, Scientific American, Ars Technica).
- **Is hard to maintain inside Python.** Editing the list means scrolling past Python syntax and re-running `--rebuild` to pick up changes for new users. Externalizing the seed to a plain text file mirrors `templates/global-claude.md` (read by `start-claude.sh:151` and `start-agent.sh:834`) and makes the file diff-friendly.

The allowlist is enforced by a tinyproxy `FilterDefaultDeny Yes` regex filter generated by `allowlist_to_regex_filter()` (`research.py:281-293`). Each non-comment line becomes `(^|\.)<domain>$`, which suffix-matches the host and any subdomain. Comments (`#`) are stripped before regex generation, so any line we add as documentation passes through untouched.

## Goals

- Allowlist seed lives in `templates/research-allowlist.txt`, loaded by `research.py` at seed time.
- Seed is organized into two clearly-labelled tiers via comment headers:
  - `# === READ-ONLY ===` (active by default)
  - `# === UPLOAD-CAPABLE — review before enabling ===` (commented out by default; user uncomments per host)
- READ-ONLY tier covers: search, reference, preprints + indexing, publishers (read paths), open data repositories, software/language docs, standards bodies, legal/regulatory, science publications, news, community, archive, major universities, government/health/stats, code hosting (read-only subdomains only), HF read CDN.
- UPLOAD-CAPABLE tier (commented out, each with a short inline rationale): `github.com`, `gitlab.com`, `bitbucket.org`, `huggingface.co`, `pypi.org` upload paths, `npmjs.com` publish, deposit hubs (`zenodo.org`, `figshare.com`, `osf.io`, `datadryad.org`), submission-capable academic platforms (`ssrn.com`, `kaggle.com`).
- A `# === EXPLICITLY EXCLUDED ===` documentation section lists domains intentionally absent (pastebin.com, webhook.site, transfer.sh, requestbin, discord webhook hosts) and why — these have no research value and are pure exfil channels.
- English-language sources only. Regional/non-English sources are out of scope (deferred — user adds per-need).
- Existing on-disk `~/.research/allowlist.txt` files are not silently overwritten. New users get the expanded seed automatically; existing users opt in via the `--reseed-allowlist` flag from Phase 3.

## Unknowns / To Verify

1. **University domain selection.** "Major universities" is a long tail. The plan should include a representative starter set (~15-20 domains) and a note that users add more via `~/.research/allowlist.txt`. Implementer should confirm the starter set covers the user's typical sources before committing — see Phase 2 step 3 for the proposed list. No external lookup required; this is a curation judgment call.
2. **Publisher submission paths.** Many publishers host both reading (`nature.com/articles/...`) and submission portals (Editorial Manager, ScholarOne) on the same hostname or close subdomains. Some submission systems are on third-party hosts (`editorialmanager.com`, `manuscriptcentral.com`) which are simply *not* in the allowlist. But where a publisher hosts submission on its own domain, hostname-only filtering can't separate it from reading. Document this in the file's preamble; do not try to engineer around it.
3. **HuggingFace download CDN scope.** Reads are served from `cdn-lfs.huggingface.co` and (sometimes) `huggingface.co` itself. Verify by inspecting a `huggingface_hub` download trace or the HF docs at `https://huggingface.co/docs/hub/storage-backends` — if read paths require the bare `huggingface.co` host, the user's tiered choice means the seed must put `huggingface.co` in UPLOAD-CAPABLE (commented out) and accept that read access is gated on the user uncommenting it. Do not silently include `huggingface.co` in READ-ONLY; that would defeat the tiering.

## Phase 3: Add `--reseed-allowlist` flag

Existing users have an on-disk `~/.research/allowlist.txt` from before the expansion and won't see the new content unless they manually delete the file. Mirror the `--reseed-global-claudemd` pattern from `start-claude.sh:26` and `start-agent.sh:90` so users can opt in.

### Steps

1. In `parse_args()` (`research.py:97-158`), add:

   ```python
   p.add_argument(
       "--reseed-allowlist",
       action="store_true",
       dest="reseed_allowlist",
       help=f"Overwrite ~/.research/allowlist.txt with the current template "
            f"({TEMPLATE_ALLOWLIST.name}). Use after pulling allowlist updates.",
   )
   ```

2. In `seed_allowlist()` (`research.py:272-276`), accept a `force: bool = False` parameter. When `force=True`, overwrite unconditionally and print "Reseeded allowlist at …" instead of "Seeded allowlist at …".

3. In `main()` (`research.py:818-875`), pass `force=args.reseed_allowlist` when calling `seed_allowlist(paths)` (`research.py:849`). The reseed should run *before* `apply_firewall()` so the new content is what gets pushed into tinyproxy in the same invocation.

4. Document the flag in the `Usage` block of the module docstring (`research.py:15-20`) and in the README.md section for `research.py` if one exists.

### Files

- `research.py` (argparse addition, `seed_allowlist` signature change, main() wiring, docstring update)
- `README.md` (if it documents `research.py` flags)

### Testing

- Existing user: `echo "junk.example.com" >> ~/.research/allowlist.txt && ./research.py --reseed-allowlist` — file is overwritten, `junk.example.com` is gone, expanded template is in place.
- Fresh user: `rm -rf ~/.research && ./research.py` — seeds from template (existing behavior, unchanged).
- `./research.py --reseed-allowlist --reload-allowlist` together — should reseed first, then reload tinyproxy with the new content. If the order is reversed, users get a confusing "reload then reseed (but the reload used the old file)" sequence.

## Notes

- **Hostname-only filtering is the load-bearing constraint.** The tiering is documentation, not enforcement. A user who uncomments `github.com` is accepting that an injected page can `POST /repos/.../issues` to it. The plan does not try to engineer around this — it makes the choice explicit.
- **No regional/non-English sources.** The user explicitly scoped this out: the local-LLM stack has no built-in translation, and adding regional sources without the user's model being good at the source language mostly produces unusable raw text. If a user later needs J-STAGE / SciELO / CNKI etc., they add the entries directly to `~/.research/allowlist.txt` — `--reseed-allowlist` would clobber them, so this is a documented trade-off, not a flaw.
- **Preserved current behavior.** `seed_allowlist()`'s "don't overwrite if exists" semantics are unchanged for the default code path. The expansion is opt-in for existing users via Phase 3 and automatic only for new users.
- **Out of scope:** changes to `start-agent.sh`'s allowlist seed (different audience, more permissive), changes to tinyproxy regex generation (`allowlist_to_regex_filter` is unchanged), changes to the iptables RESEARCH chain. The firewall plumbing already handles arbitrarily long allowlists.
- **Curation will keep moving.** Treat `templates/research-allowlist.txt` as living configuration, not as code. Edits to add or remove a domain should be one-line PRs reviewable in seconds. That ergonomic is the whole point of externalizing it.
