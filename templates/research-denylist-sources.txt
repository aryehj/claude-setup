# research.py upstream denylist feeds — pinned by commit SHA.
#
# Apply changes:
#   ./research.py --refresh-denylist     fetch fresh copies from currently-pinned URLs
#   ./research.py --reseed-denylist      overwrite this file from the repo template
#                                        (use after pulling start-claude updates)
#
# Format: one URL per line; '#' starts a comment.
#
# Pinning policy: never use 'main' / 'HEAD'. Pinning to a specific SHA forces
# a human review of upstream changes before they take effect locally — both
# routine churn and a hypothetical compromise of the upstream repo. To pick up
# upstream changes, edit this file (or run --reseed-denylist after the repo
# template is updated) and then run --refresh-denylist.
#
# Feed format: `wildcard/<list>-onlydomains.txt` — one apex/registrable domain
# per line, with subdomain hierarchies pre-rolled-up by hagezi. This is the
# right shape for Squid `dstdomain`: research.py prefixes each entry with `.`
# at compose time, so a single `.foo.com` entry covers the apex AND every
# subdomain. The older `domains/<list>.txt` files list subdomains exhaustively
# but omit the apex, which causes Squid to leak `https://foo.com/` while
# blocking `https://www.foo.com/`. The onlydomains variants are also ~50%
# smaller.
#
# hagezi/dns-blocklists, pinned 2026-04-25
#   pro-onlydomains      ~183k entries  Ads / tracking / scam / malware / fake (broad)
#   fake-onlydomains     ~14k entries   Fake stores, scam streaming, rip-offs
#   tif-onlydomains      ~600k entries  Threat intel feed (rotates frequently)
#
# All three feeds are enabled by default. Verified at the default VM size:
# combined ~800k apex entries, Squid worker RSS ~100 MB, parse ~5 s. If you
# add further feeds, re-check headroom with `ps -eo pid,rss,cmd | grep [s]quid`
# and bump `./research.py --rebuild --memory=4` (or larger) if needed.
